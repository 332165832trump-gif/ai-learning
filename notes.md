## 技术笔记

### 1. Git 基础操作

#### 1.1 初始化仓库 — `git init`

**作用**：把普通文件夹变成 Git 仓库（让 Git 开始跟踪这个文件夹里的变化）。

```bash
# 在项目文件夹里执行
git init
# 输出：Initialized empty Git repository in ...
```

**发生了什么**：Git 在文件夹里悄悄创建了一个隐藏目录 `.git`，所有的版本历史、快照信息都存在里面。删除 `.git` = 仓库变回普通文件夹。

**什么时候用**：新项目的第一条命令。只执行一次。

---

#### 1.2 查看状态 — `git status`

**作用**：看当前仓库里什么文件改过、什么文件没被跟踪。

```bash
git status
# 红色文件名 = 改过但还没 add
# 绿色文件名 = 已经 add，准备 commit
# 提示 "nothing to commit" = 没有新变化
```

**本质**：这是一个"只读"命令，不会改任何东西。它是你的安全绳——每次动手前先看一眼，心里有数。

**新手习惯**：`git add` 之前和 `git commit` 之前各跑一次，养成肌肉记忆。

---

#### 1.3 暂存改动 — `git add`

**作用**：告诉 Git "这些文件的改动我要放到下一个 commit 里"。

```bash
# 暂存单个文件
git add classifier.py

# 暂存当前文件夹所有改动
git add .
```

**本质**：Git 的 commit 不是自动拍所有改动，而是只拍你"add 过的"。这让你可以把一个大改动拆成多个逻辑清晰的 commit。

**关键对比**：

| 状态 | 含义 |
|---|---|
| Working Directory | 你正在改的文件（还没 add） |
| Staging Area | 你 add 过的文件（准备 commit） |
| Repository | 你 commit 过的文件（已保存到历史） |

---

#### 1.4 提交版本 — `git commit -m "..."`

**作用**：把你 add 过的改动拍成一个快照，永久保存到仓库历史里。

```bash
git commit -m "add file classifier"
```

- `-m`：message（消息），引号里写这次改了什么的说明。
- 不加 `-m`，Git 会弹出一个编辑器让你写消息（初学者直接用 `-m` 更简单）。

**工程规范**：
- 消息用英文，动词开头，小写。
- 好例子：`"add user login feature"`
- 烂例子：`"改了点东西"`、`"fix"`

---

#### 1.5 关联远程仓库 — `git remote add origin <URL>`

**作用**：告诉本地仓库，你的代码要推送到 GitHub 上哪个仓库。

```bash
git remote add origin https://github.com/16068/oc-test.git
```

- `origin`：远程仓库的昵称（约定俗成用 origin，你也可以叫别的）。
- 这条命令只执行一次。

**通俗类比**：本地仓库 = 你的电脑，`git remote add` = 给它一个 GitHub 的"收货地址"。之后 push 就是往这个地址发货。

---

#### 1.6 推送到远程 — `git push -u origin main`

**作用**：把本地 commit 上传到 GitHub。

```bash
git push -u origin main
```

- `-u`：把本地 `main` 分支和远程 `origin/main` 绑定。绑定后下次直接 `git push` 就行，不用再写 `origin main`。
- `origin`：远程仓库名。
- `main`：分支名。

**新手注意**：push 之前必须至少有一次 commit，否则 Git 说"没什么可推的"。

---

#### 1.7 完整工作流总结

```
1. git init              ← 仓库初始化（只做一次）
2. git remote add ...    ← 关联 GitHub（只做一次）
3. 写代码 / 改文件 ...
4. git status            ← 看看改了什么
5. git add .             ← 暂存改动
6. git commit -m "..."   ← 提交快照
7. git push              ← 推上 GitHub
8. 回到第 3 步，循环
```

### 2. Python 基础

#### 2.1 输入输出 — `input()` 和 `print()`

```python
name = input("请输入你的名字: ")    # 等待用户输入，返回字符串
print("Hello " + name)              # 输出拼接后的字符串
```

- `input()` 返回的**永远是字符串**，即使用户输入 `25`，`name` 里也是 `"25"`（文字），不是数字。
- `+` 在字符串之间是拼接，在数字之间是加法。不能混用。
- `print()` 不会自动加空格，拼接时空格要自己加（`"Hello "` 末尾有空格）。

#### 2.2 导入模块 — `import os`

```python
import os
```

- `import` = "把这个工具箱拿进来"。
- `os` 是 Python 自带的操作系统工具模块（不需要安装）。
- 用模块里的函数必须加前缀：`os.listdir()`，光写 `listdir()` 会报 `NameError`。

#### 2.3 列出文件 — `os.listdir(path)`

```python
files = os.listdir(".")   # "." = 当前目录
print(files)              # 输出：['classifier.py', 'main.py', '.git']
```

- 返回一个**列表**，包含该路径下所有文件和文件夹的名字。
- `.` 表示当前目录，`..` 表示上一级目录。
- 路径不存在 → `FileNotFoundError`。

#### 2.4 Windows 路径 + Python 字符串 = SyntaxError

**报错信息**：
```
SyntaxError: (unicode error) 'unicodeescape' codec can't decode bytes
```

**原因**：反斜杠 `\` 是 Python 字符串的转义字符。`\n` 是换行，`\t` 是 Tab。`C:\Users` 里的 `\U` 被当成 Unicode 转义 → 崩溃。

**3 种正确写法**：
```python
# 方法 1：用正斜杠（最推荐）
path = "C:/Users/16068/桌面"

# 方法 2：双反斜杠
path = "C:\\Users\\16068\\桌面"

# 方法 3：原始字符串
path = r"C:\Users\16068\桌面"
```

**最佳实践**：Python 代码里永远用 `/` 写路径；用户通过 `input()` 输入的路径天然安全，不会触发转义。

#### 2.5 Python 缩进的作用

**我遇到的问题**：不理解 `for`、`if`、`print` 的缩进关系，不知道缩进的作用。

**代码示例**：
```python
for file in files:
    if file.endswith(".py"):
        print(file + " 是代码文件")
    else:
        print(file + " 是其他文件")
```

**错误示例**：
```python
for file in files:
    if file.endswith(".py"):
print(file + " 是代码文件")    # 没缩进
```

**错误原因**：`IndentationError: expected an indented block`。Python 用缩进判断"这个 `print` 属于 `if` 还是属于 `for`"。`if` 后面需要至少一行缩进的代码，但你跳到了行首，Python 不知道你想让它执行什么。

**正确理解（重点）**：

- **缩进 = 层级结构**：缩进越多，所属层级越深。就像大纲：
  ```
  for                         → 第 1 层
      if                      → 第 2 层（属于 for）
          print("A")          → 第 3 层（属于 if）
  ```

- **决定代码属于哪个逻辑块**：
  ```python
  for file in files:          # 循环体开始
      print(file)             # 属于 for，每轮都执行
      if file.endswith(".py"): # 属于 for
          print("代码")       # 属于 if，条件成立才执行
  print("循环结束")            # 没有缩进，不属于 for，只执行一次
  ```

- **控制执行范围**：缩进 = 冒号下面"谁听谁的"。缩进 4 格是 Python 约定，Tab 也行但别混用。

**我的理解**：缩进就是"代码的身份证"，告诉 Python "我属于谁"。`if` 下面缩进的代码 = `if` 的小弟，条件成立时才干活。不缩进的代码 = 自由人，不受任何条件约束，每次都执行。

**下次如何避免**：
- 任何以 `:` 结尾的行（`for`、`if`、`elif`、`else`），下一行必须缩进 4 个空格。
- 同一层级的代码缩进格数必须一致。
- 写完 `:` 按回车，大多数编辑器自动帮你缩进。

#### 2.6 文件类型判断 — `endswith()` + `for` + `if/elif/else`

**代码（classifier.py 当前版本）**：
```python
import os

path = input("请输入文件夹路径: ")

files = os.listdir(path)

for file in files:
    if file.endswith(".jpg") or file.endswith(".png") or file.endswith(".gif"):
        print(file + " → 图片")
    elif file.endswith(".pdf") or file.endswith(".txt") or file.endswith(".docx"):
        print(file + " → 文档")
    elif file.endswith(".py"):
        print(file + " → 代码")
    else:
        print(file + " → 其他")
```

**关键知识点**：

- **`str.endswith("尾巴")`**：判断字符串是否以指定内容结尾，返回 `True` 或 `False`。
- **`or`**：逻辑"或"。`a or b` → 两者有一个是 `True` 就通过。
- **`if` / `elif` / `else` 是排队判断**：从上到下，第一个满足的就执行，后面全部跳过。`else` 是兜底。
- **`elif` 不是 `else if`**：Python 里必须连写成 `elif`，空格会报 `SyntaxError`。

**常见错误**：

| 错误写法 | 后果 |
|---|---|
| `if .endswith(".jpg")` 忘写 `file` | `NameError` |
| `elif` 写成 `else if` | `SyntaxError` |
| `else:` 后面还写条件 | `SyntaxError` |
| `file.endswith(".jpg" or ".png")` | 语法不报错，但逻辑错误——只检查 `.jpg` |

| 语法 | 效果 |
|---|---|
| `# 标题` | 一级标题 |
| `## 标题` | 二级标题 |
| `**粗体**` | **粗体** |
| `` `代码` `` | 行内代码 |
| ` ``` ... ``` ` | 代码块 |
| `- 条目` | 无序列表 |
| `| 列1 | 列2 |` | 表格 |
| `---` | 分割线 |

### 1.8 分支与协作（从单人到团队）

#### 为什么要用分支

当你开始做稍微复杂一点的项目时，不建议在 `main` 上直接改。更好的方式是：

- `main` 只放稳定可用代码
- 新功能在新分支开发
- 测试通过后再合并回 `main`

#### 常用命令

```bash
# 查看本地分支
git branch

# 创建并切换到新分支
git checkout -b feature/price-monitor

# 或者新语法（推荐）
git switch -c feature/price-monitor

# 切回主分支
git switch main

# 合并分支（在 main 上执行）
git merge feature/price-monitor
```

#### 分支命名建议

- `feature/...`：新功能
- `fix/...`：修 bug
- `refactor/...`：重构
- `docs/...`：文档更新

示例：
- `feature/course-alert`
- `fix/login-timeout`

---

#### Pull Request（PR）是什么

PR 是「把你分支上的改动，申请合并到主分支」的流程。它的价值不只是“点个按钮”，而是：

1. 让改动可审阅（别人能看出你具体改了什么）
2. 让风险可控（CI 测试通过才合并）
3. 让知识可沉淀（讨论记录可追溯）

**写 PR 描述的最小模板**：

- 背景：为什么要改
- 变更：改了哪些点
- 验证：怎么测的，结果是什么
- 风险：已知副作用/兼容性影响

### 1.9 Git 进阶排错

#### `git diff`：看“具体改了哪几行”

```bash
# 看还没 add 的改动
git diff

# 看已经 add、准备 commit 的改动
git diff --staged
```

这是防止“误提交”的关键命令。commit 前至少看一遍 diff。

#### `git log`：看历史提交

```bash
# 紧凑日志
git log --oneline --graph --decorate -20
```

推荐别名（可选）：

```bash
git config --global alias.lg "log --oneline --graph --decorate"
```

之后用 `git lg` 就能快速查看历史结构。

---

#### 撤销操作（高频且重要）

```bash
# 取消暂存（把 add 撤回）
git restore --staged <file>

# 丢弃工作区改动（慎用，会丢失未提交内容）
git restore <file>

# 回到上一个提交（保留改动在工作区）
git reset --soft HEAD~1
```

> 原则：不确定时先 `git status` + `git diff`，确认后再撤销。

### 2. Python 自动化工程化补充

#### 2.1 项目目录建议

当脚本数量变多后，建议按职责拆目录：

```text
project/
  data/            # 原始与处理后数据
  logs/            # 日志文件
  scripts/         # 自动化脚本入口
  src/             # 核心业务代码（函数、类）
  tests/           # 测试代码
  config/          # 配置（yaml/env）
  requirements.txt # 依赖清单
```

这样做的好处是：代码、数据、日志职责分离，后期维护难度会显著下降。

---

#### 2.2 环境管理（避免“我这能跑，你那不能跑”）

```bash
# 创建虚拟环境
python -m venv .venv

# 激活（macOS/Linux）
source .venv/bin/activate

# 安装依赖
pip install -r requirements.txt

# 导出依赖
pip freeze > requirements.txt
```

核心目标：保证不同机器上依赖版本一致，降低环境问题。

---

#### 2.3 配置与密钥管理

- 账号密码、API Key 不要写死在代码里。
- 使用 `.env` 或系统环境变量。
- `.env` 必须加入 `.gitignore`。

示例（伪代码思路）：

- `os.getenv("API_KEY")` 读取密钥
- 代码中只处理“密钥缺失时的报错提示”

#### 2.4 日志规范（代替 print）

推荐最小日志字段：

- 时间

实践建议：

### 3. 数据采集的“稳健性”设计

#### 3.1 反脆弱采集策略

不要写“一次请求成功才算完成”的脆弱脚本，至少考虑：

- 超时控制（timeout）
- 重试机制（指数退避）
- 失败记录（写入失败队列）
- 幂等保存（重复运行不会产生脏数据）

#### 3.2 数据质量检查（比采集本身更重要）

每次入库前都做 3 件事：

1. 行数检查：是否异常暴涨/暴跌
2. 主键去重：如 `date + symbol`
3. 空值检查：关键字段是否缺失

如果检查失败，不要静默继续，应该报警并标记该批次数据不可用。

### 4. 金融建模学习的正确顺序（避免一上来就“调参”）

#### 4.1 五步法

1. **定义问题**：你想解决什么投资问题（择时/选股/风控）
2. **构造假设**：为什么这个因子可能有效
3. **回测验证**：收益、回撤、换手、夏普
4. **稳健性检验**：不同市场阶段是否都有效
5. **实盘约束**：滑点、手续费、容量、交易限制

#### 4.2 常见误区

- 只看收益率，不看最大回撤
- 在单一时间区间调参到极致（过拟合）
- 忽略交易成本，导致“纸面盈利”
- 用未来数据污染历史（未来函数）

### 5. AI 协作开发方法（你当前路线的关键）

#### 5.1 提问模板（给 GPT/编程助手）

高质量提示词结构：

1. **目标**：要实现什么
2. **输入/输出**：数据格式和示例

#### 5.2 代码验收清单（AI 生成代码必须过这一关）

- 是否可运行（依赖完整）
- 是否可读（命名清晰、函数职责单一）
- 是否可测（关键逻辑有测试样例）
- 是否可监控（有日志与错误处理）
- 是否合规（不触碰平台禁止行为）

### 6. 可执行学习计划（8 周版）

#### 第 1-2 周：Python + Git 基础巩固

- 每天 1 个小脚本（文件、CSV、请求、解析）
- 每天至少 1 次完整 Git 流程（add/commit/push）
- 输出：`scripts/` 下累计 10~14 个可运行脚本

#### 第 3-4 周：数据采集项目化

- 选 1 个公开数据源（例如基金净值/公告）
- 完成：抓取 → 清洗 → 落盘 → 日志 → 异常重试
- 输出：一个可定时运行的采集器

#### 第 5-6 周：自动化监控

- 做“网页状态变化提醒”项目
- 增加：邮件/企业微信/桌面通知之一
- 输出：可连续运行的监控与提醒系统

#### 第 7-8 周：金融分析闭环

- 读取历史数据，完成至少 1 个策略回测
- 指标最少包含：年化收益、最大回撤、夏普比率
- 输出：图表 + 简短策略复盘报告

### 7. 你的长期壁垒（总结）

短期看，你在学的是 Python、爬虫、自动化；
长期看，你真正构建的是以下壁垒：

1. **业务抽象能力**：把模糊需求拆成可执行流程
2. **系统化能力**：从脚本到稳定服务的工程思维
3. **风险控制能力**：合规意识 + 数据质量意识
4. **AI 协作能力**：把 AI 变成放大器，而不是玩具

当这四个能力叠加，你就不只是“会写脚本的人”，而是能持续交付结果的自动化工程师。
